# 侧边栏文件管理 — 方案规划

## 现有架构分析

### 代码结构

项目是零框架的 vanilla JS 单页应用，仅一个源文件 `src/main.js`（~670 行），通过 Vite 构建。入口函数 `init()` 在 `window.load` 时调用。

```
src/main.js          ← 所有逻辑（编辑器、预览、Mermaid、主题、持久化）
index.html           ← HTML 骨架（header + 左右分栏 + 分割线）
public/css/style.css ← 全局样式 + 暗色主题
```

### 核心流程

1. Monaco Editor 加载 → `convert()` → marked 解析 → DOMPurify 净化 → 注入 `#output`
2. Mermaid 图表：`convert()` 输出 `<pre class="mermaid">` 占位 → 150ms 防抖后 `mermaid.render()` 异步渲染为 SVG
3. 分栏拖拽：`#split-divider` 监听 mousedown/move/up 调整左右 pane 宽度
4. 状态持久化：通过 Storehouse-js 存储 3 个 key（编辑器内容、滚动同步、主题设置）

### 关键约束

- Monaco Editor 从 CDN 加载（jsdelivr ES module），不走 Vite 打包
- 没有 vite.config，使用 Vite 默认配置
- 所有 DOM 操作都是手动 `document.querySelector` + `addEventListener` 模式
- 没有模块化——所有函数在一个文件内通过闭包共享状态

### 改造要点

- 需要在原有左右分栏基础上，在最左侧新增侧边栏面板
- 需要将代码模块化拆分，避免 `main.js` 过度膨胀
- 需保持与现有主题系统（`data-theme` 属性、暗色/亮色切换）的兼容
- 多文件模型需要改造编辑器的"当前内容"概念

## 依赖调研

### 1. 文件树组件选型

核心需求：vanilla JS 可用 + 支持拖拽排序 + 高下载量/稳定。

| 库 | 周下载量 | 框架 | 拖拽 | 问题 |
|---|---|---|---|---|
| **SortableJS** | 2.1M | Vanilla | 原生支持 | 仅为 D&D 引擎，无树 UI，需自建渲染 |
| jsTree | ~25K/周 | jQuery | 支持 | 依赖 jQuery（~87KB），过于臃肿 |
| fancytree | ~12K/周 | jQuery | 支持 | 依赖 jQuery + jQuery UI |
| sortable-tree | ~低数百 | Vanilla TS | 原生支持 | 功能完全匹配但下载量极低 |
| @webreflection/file-tree | 极低 | Web Component | 不支持 | 无拖拽能力 |

**结论：不存在同时满足"vanilla JS + 内置拖拽 + 高下载量"三条件的现成树组件。** 这是 npm 生态的一个空白地带——高质量树组件几乎全部绑定 React/Vue 框架。

#### 推荐方案：SortableJS + 自定义树渲染

**SortableJS 作为 D&D 引擎**（2.1M 周下载、31K GitHub Stars、MIT 协议）：
- 原生支持嵌套列表拖拽（`group` 选项），与文件/目录层级结构天然匹配
- 提供 `onEnd` 回调，可精确获知拖动后的新位置和新父级
- 体积极小（~4KB gzipped），零依赖
- 已在 VSCode、Trello、Asana 等产品中验证

**自定义树渲染器**（~100-150 行 vanilla JS）：
- 渲染为嵌套 `<ul>/<li>` 结构，每行包含图标 + 名称
- 支持单击选中/打开文件、双击重命名
- 通过 CSS class 控制展开/收起、选中态、拖拽悬停态
- 与 SortableJS 配合：每个 `<ul>` 作为一个 Sortable 实例（`group: 'files'`），实现跨层级拖拽

**备选方案（如果用户坚持使用成品树组件）：sortable-tree**
- 功能完全匹配需求（vanilla TS + D&D + 折叠 + 持久化），但周下载量极低
- 风险：项目较新，可能存在未发现的 bug，社区支持弱

#### 各 npm 包体量估算

| 包 | 大小（gzipped） | 说明 |
|---|---|---|
| sortablejs | ~4KB | 仅 D&D 引擎 |
| localforage | ~3KB | IndexedDB 封装 |
| lucide | ~1KB + 按需图标 | tree-shaking 后仅加载使用到的图标 |
| **合计** | **~8KB** | 对现有应用体量影响极小 |

### 2. lucide 图标

`lucide` 是 lucide-icons 的 vanilla JS 包，支持 tree-shaking。

使用方式（Vite 环境下）：
```js
import { createIcons, FilePlus, FolderPlus, Folders } from 'lucide';
// 在 HTML 中使用 <i data-lucide="file-plus"></i>
createIcons({ icons: { FilePlus, FolderPlus, Folders } });
```

需要的 3 个图标：
- `FilePlus` — 新增文件
- `FolderPlus` — 新增目录
- `Folders` 或 `FolderSync` — 展开/收起全部（`FolderSync` 视觉上更适合"同步展开状态"的概念）

### 3. localforage

- 周下载 ~4M，IndexedDB 优先的异步存储，API 与 localStorage 类似
- 自动降级：IndexedDB → WebSQL → localStorage
- 存储容量远大于 localStorage（250MB+ vs 5-10MB）
- 异步非阻塞，适合存储 Markdown 文件内容

## 数据模型设计

### 存储结构

localforage 实例配置：

```js
const store = localforage.createInstance({
  name: 'markdownLivePreview',
  storeName: 'filesystem'
});
```

### 单一 Key 存储方案

使用一个 key `filesystem:state` 存储完整的文件系统状态：

```ts
interface FileSystemState {
  nodes: Record<NodeId, FileNode>;  // 扁平 Map，O(1) 按 ID 查找
  rootChildren: NodeId[];            // 根级节点有序列表
  activeFileId: NodeId | null;       // 当前在编辑器中打开的文件 ID
}

interface FileNode {
  id: string;           // UUID v4
  name: string;         // 文件/目录名
  type: 'file' | 'directory';
  parentId: string | null;  // 父目录 ID，null 表示根级
  children: string[];   // 子节点有序列表（目录专用，文件为空数组）
  content: string;      // Markdown 内容（文件专用，目录为空字符串）
  createdAt: number;    // Date.now()
  updatedAt: number;    // Date.now()
}
```

#### 设计理由

- **扁平 Map + children 数组**：比嵌套 tree 更容易做拖拽排序（只需操作 children 数组顺序）
- **单 key 存储**：读写都是一次 IndexedDB 事务，保证原子性
- **content 内联**：Markdown 文件通常 < 100KB，无需分离存储
- **parentId 冗余**：方便向上遍历路径（面包屑导航可能用到）

#### 默认初始状态

```js
const defaultState = {
  nodes: {
    'welcome': {
      id: 'welcome',
      name: 'Welcome.md',
      type: 'file',
      parentId: null,
      children: [],
      content: defaultInput,  // 现有的默认 Markdown 示例内容
      createdAt: Date.now(),
      updatedAt: Date.now()
    }
  },
  rootChildren: ['welcome'],
  activeFileId: 'welcome'
};
```

### 侧边栏内部布局

```
┌──────────────────┐
│ [+] [📁+] [⊞]   │  ← 顶部按钮栏（3 个 lucide 图标，无文字）
├──────────────────┤
│ 📁 folder1     ▸ │  ← 目录项（可展开/收起、可拖拽）
│   📄 file1.md    │  ← 文件项（可拖拽、点击打开）
│   📄 file2.md    │  ← 选中文件高亮背景
│ 📁 folder2     ▸ │
│   📄 file3.md    │
│ 📄 README.md     │  ← 根级文件
└──────────────────┘
```

### 交互行为

| 操作 | 行为 |
|---|---|
| 点击 `+` 按钮 | 在根级创建新文件 `Untitled.md`，自动选中并聚焦名称输入 |
| 点击 `📁+` 按钮 | 在根级创建新目录 `New Folder`，自动聚焦名称输入 |
| 点击 `⊞` 按钮 | 递归展开或收起所有目录（toggle 模式） |
| 单击文件 | 切换 `activeFileId`，加载内容到编辑器，自动保存当前文件 |
| 双击文件/目录名 | 进入重命名模式（inline input） |
| 拖拽节点 | 通过 SortableJS 调整位置/层级，自动更新 parentId 和 children 数组 |
| 右键节点 | 弹出右键菜单（重命名、删除、新建子文件/子目录） |

## 代码模块化方案

### 文件拆分

```
src/
├── main.js          ← 入口：组装各模块，初始化应用
├── storage.js       ← localforage 封装：load/save/CRUD
├── sidebar.js       ← 侧边栏 UI：渲染树、按钮事件、右键菜单
├── tree-renderer.js ← 树 DOM 渲染 + SortableJS 集成
├── editor.js        ← Monaco Editor 初始化 + 内容变更事件
└── constants.js     ← 默认内容、CSS class 名常量
```

### 模块职责

**`storage.js`** — 纯数据层
- `loadFileSystem()` → FileSystemState
- `saveFileSystem(state)` → void
- `createFile(parentId, name)` → FileNode
- `createDirectory(parentId, name)` → FileNode
- `deleteNode(id)` → void
- `renameNode(id, newName)` → void
- `moveNode(id, newParentId, newIndex)` → void
- `getNodePath(id)` → string[] （面包屑路径）

**`tree-renderer.js`** — DOM 渲染 + D&D
- `renderTree(state, container)` → 渲染整个树
- `renderNode(node)` → HTMLElement
- `refreshNode(id)` → 局部更新单个节点
- SortableJS 实例化：每个 `<ul>` 一个 Sortable，通过 `group: 'sidebar-files'` 互联
- `onSortEnd` 回调 → 调用 `storage.moveNode()`

**`sidebar.js`** — UI 控制器
- 按钮事件绑定（新建文件/目录、展开/收起）
- 节点点击/双击事件委托
- 右键菜单
- 内联重命名逻辑
- 与 editor 模块协调（切换文件时先保存再加载）

**`editor.js`** — 编辑器模块（从 main.js 抽取）
- `setupEditor(container)` → editor 实例
- `setContent(content)` → 设置编辑器内容
- `getContent()` → 获取当前内容
- `onChange(callback)` → 内容变更回调

### 模块间通信

```
sidebar.js ──(文件切换)──→ editor.js ──(内容变更)──→ storage.js
    │                          │
    └──(树结构变更)──→ storage.js ──(持久化)──→ localforage
                           │
tree-renderer.js ──(D&D完成)──→ sidebar.js
```

使用简单的回调函数传递，不引入事件总线（保持简洁，与现有代码风格一致）。

## 风险与边界处理

| 场景 | 处理方式 |
|---|---|
| 第一次打开（无存储数据） | 创建默认 Welcome.md 文件 |
| 同名文件/目录 | 自动追加数字后缀 `(1)`, `(2)` |
| 删除已打开的文件 | 切换到第一个可用文件，若无则显示空白 |
| 文件内容为空 | 正常显示，编辑器显示空内容 |
| 拖拽文件到文件内部 | 禁止，仅目录可接受子节点 |
| 根级拖拽限制 | 通过 `lockRootLevel` 或自建校验防止非法操作 |
| 大文件内容（>1MB） | 不做限制（IndexedDB 兼容），但提示用户 |
| 存储空间不足 | catch `QuotaExceededError`，提示用户清理 |
